## WICHTIG: Kontext und Abgrenzung

### Was ist das?

Ein **postalisch versendeter Print-Onepager** fuer Endkunden, der die Kernbotschaften der bestehenden ARM-Landing-Page auf ein physisches DIN-A4-Format verdichtet. Aufhaenger: Frostschaeden-Saison / Fruehjahrsinstandsetzung.

### Was ist es NICHT?

- Keine Landing Page (kein Scrollen, kein Interaktionsdesign)
- Keine Broschuere (maximal 2 Seiten)
- Kein Datenblatt (kein reines Faktenblatt, sondern Storytelling + Fakten)
- Kein Aktionsflyer (kein Rabattangebot, keine zeitliche Begrenzung)

### Verhaeltnis zur Landing Page

Der Onepager ist das **postalische Pendant** zur Landing Page v2. Er erzaehlt dieselbe Geschichte -- radikal gekuerzt. Der QR-Code fuehrt zur Landing Page fuer alle, die mehr wissen wollen.

```
Postversand → Onepager im Briefkasten
  → Interesse geweckt
    → QR-Code scannen → Landing Page lesen
      → Beim Haendler bestellen
```

## 2. Zielgruppe

**Endkunden direkt** (identisch zur Landing Page v2):

| Zielgruppe | Kernmotivation |
|------------|---------------|
| Kommunen / Tiefbauaemter | Frostschaeden-Saison: Budget effizient nutzen, jede Stelle nur einmal reparieren |
| GaLa-Bau / Strassenbau | Zuverlaessiges Material fuer Kundenprojekte, keine Reklamationen |
| Facility Management | Schlagloecher beseitigen, Haftungsrisiko minimieren, laufender Betrieb |
| Verarbeiter / Handwerker | Einfach verarbeiten, professionelle Ergebnisse ohne Spezialgeraete |

**Einstiegssituation:** Der Empfaenger oeffnet seinen Briefkasten und findet den Onepager. Er hat maximal 10--15 Sekunden Aufmerksamkeit, bevor er entscheidet: lesen oder wegwerfen. Der Onepager muss in dieser Zeit einen klaren Nutzen kommunizieren.

---

## 3. Aufhaenger: Frostschaeden-Saison

Das saisonale Timing ist der emotionale Einstieg:

- **Maerz/April:** Frostperiode vorbei, Schaeden werden sichtbar
- **Pain Point:** "Jedes Fruehjahr dasselbe Bild: Schlagloecher, Risse, Absackungen"
- **Loesungsversprechen:** "Dieses Jahr: Einmal reparieren. Dauerhaft halten."

Der Frostschaeden-Aufhaenger ist **kein Aktionsangebot**, sondern ein **Problem-Timing** -- wir treffen den Empfaenger im Moment seines Schmerzpunkts.

## 4. Conversion-Ziel

**Der Onepager hat EIN Ziel:** Den Empfaenger dazu bringen, den QR-Code zu scannen und die Landing Page zu lesen.

- **Primaer:** QR-Code zur Landing Page scannen
- **Sekundaer:** KORODUR im Kopf behalten fuer den naechsten Haendlerkontakt
- **Kein Ziel:** Kein Formular, kein Direktkauf, kein Anruf-CTA als Hauptziel

### QR-Code

- **URL:** `https://sfleischmann-3steps2.github.io/Marketing-Team-Skills/projekte/korodur-asphalt-repair-mix/landing-page/index.html`
- **Platzierung:** Prominent, aber nicht dominant. Neben einem klaren Handlungshinweis.
- **Begleittext:** "Alle Fakten, Referenzprojekte und das technische Datenblatt:" oder "Mehr erfahren unter:"
- **Groesse:** Mindestens 25 x 25 mm fuer zuverlaessiges Scannen

## 6. Corporate Design (Print-Anpassungen)

Alle Werte aus dem KORODUR Corporate Design gelten. Fuer Print sind folgende Anpassungen relevant:

### Farben

| Farbe | Hex (Screen) | CMYK (Print) | Einsatz im Onepager |
|-------|-------------|-------------|---------------------|
| Dunkelblau | #002d59 | 100/70/8/54 | Headlines, Textfarbe, Rahmen |
| Cyan | #009ee3 | 100/0/0/0 | Key Figures, Highlights, QR-Code-Rahmen |
| Hellgrau | #ececed | 0/0/0/10 | Hintergrundflaechen (Tabelle, Referenzen) |
| Mittelgrau | #d9dada | 0/0/0/20 | Trennlinien |
| Weiss | #ffffff | -- | Standard-Hintergrund |

### Typografie

- **Gabarito** als Hauptschrift (wie digital)
- **Fuer Print-PDF-Export:** Gabarito wird via Google Fonts geladen; im HTML-Print-CSS als `@media print` eingebunden
- Headlines: Gabarito Black, VERSAL
- Fliesstext: Gabarito Regular, 9--10 pt (Print-aequivalent)
- **Keine Schrift kleiner als 8 pt** (Lesbarkeit im Druck)

### Logo

- KORODUR Logo: Farbige Variante auf weissem Hintergrund
- Platzierung: Links oben (wie im Header der Landing Page)
- Schutzraum beachten (Hoehe des "O" aus dem Logoschriftzug)
- Logo-Datei: `assets/korodur-logo.png`

### Sonstiges

- Gedankenstriche: En-Dash (--), nicht Em-Dash
- Tabellen: Kennwerte fett, Werte zentriert
- Buttons gibt es nicht im Print -- stattdessen QR-Code als "Call to Action"

## 7. Technische Anforderungen

### Lieferformat

- **HTML-Datei** mit eingebettetem CSS (wie Landing Page)
- **@media print** Stylesheet fuer sauberen PDF-Export aus dem Browser
- **@page** Regel fuer DIN A4 Format (210 x 297 mm)
- Variante A: 1 Seite (`@page { size: A4; }`)
- Variante B: 2 Seiten mit `page-break-after: always` nach Vorderseite

### Print-CSS Anforderungen

```css
@media print {
  @page {
    size: A4;
    margin: 12mm 15mm;
  }
  body { -webkit-print-color-adjust: exact; print-color-adjust: exact; }
  /* Hintergrundfarben im Druck erzwingen */
}
```

### Bildmaterial

Verfuegbare Assets (aus dem Projektordner `assets/`):
- `korodur-logo.png` -- KORODUR Logo
- `Asphalt Repair Mix_30 Minuten.png` -- Produktbild (25-kg-Sack)

- Generierung: Inline als SVG oder via JavaScript-Bibliothek (z.B. `qrcode.js`)

- Mindestgroesse: 25 x 25 mm

- Sachlich-informativ, nicht werblich-dringlich

- **Keine Fragen als Headlines** (im Print wirken rhetorische Fragen weniger gut)
- **Aktive Formulierungen** bevorzugen: "ARM haelt dauerhaft" statt "Warum Kaltmischgut nicht reicht"
- **Zahlen hervorheben:** Key Figures visuell prominent (grosse Ziffern, Cyan)

## 10. Print-Design-Prinzipien

### Blickfuehrung

Der Onepager muss in einer klaren Reihenfolge gelesen werden:

1. **Logo + Headline** (oben links/mittig) -- Absender + Versprechen
2. **Key Figures** -- Sofortiger Beweis
3. **Problem/Loesung** -- Warum relevant, wie funktioniert's
4. **Referenz + QR-Code** -- Beweis + naechster Schritt

### Weissraum

- Print braucht MEHR Weissraum als digital
- Nicht alles vollpacken -- lieber weniger Inhalt, dafuer lesbar
- Mindestens 15 mm Seitenrand (Druckbereich)
- Absaetze klar trennen (Abstand oder Linien)

### Kontrast

- Text immer Dunkelblau auf Weiss/Hellgrau (maximaler Kontrast)
- Cyan sparsam einsetzen (Highlights, Zahlen, Akzente)
- Keine helle Schrift auf hellem Hintergrund

### Druckqualitaet

- Bilder: Mindestens 150 dpi bei Ausgabegroesse (fuer Web-Assets ggf. skalieren)
- QR-Code: Vektorgrafik (SVG) bevorzugen
- Farben: CMYK-Werte im CD definiert, Browser-Print kann RGB-zu-CMYK-Konvertierung uebernehmen

## 11. Workflow und Phasenplanung

### Beteiligte Agents

| Agent | Aufgabe | Prioritaet |
|-------|---------|-----------|
| **Agent 1 (Stratege)** | Kernbotschaften fuer Print destillieren, Storytelling-Bogen auf 1--2 Seiten, Blickfuehrung planen | Hoch |
| **Agent 2 (Content Creator)** | Texte schreiben -- beide Varianten, radikal gekuerzt, Print-optimiert | Hoch |
| **Agent 4 (CRO-Optimierer)** | Blickfuehrung validieren, QR-Code-Platzierung, Print-"Conversion" optimieren | Mittel |
| **Agent 6 (Builder)** | HTML + Print-CSS Umsetzung, beide Varianten, PDF-Export-tauglich | Hoch |

**Nicht beteiligt:**
- Agent 3 (SEO) -- Print hat kein SEO
- Agent 5 (Growth) -- Versandplanung liegt beim Nutzer

### Phasen

```
Phase 1: Strategie (Agent 1)
  → Kernbotschaften fuer Print ableiten
  → Blickfuehrung / visuelles Storytelling definieren
  → Entscheidung: Welche Inhalte auf welche Variante

Phase 2: Content (Agent 2)
  → Texte fuer Variante A (einseitig)
  → Texte fuer Variante B (zweiseitig)
  → Jeweils mit konkreten Laengenangaben (Zeichen/Woerter)

Phase 3: CRO-Review (Agent 4)
  → Blickfuehrung pruefen
  → QR-Code-Platzierung und -Groesse
  → Print-UX: Wird die Seite in der richtigen Reihenfolge gelesen?

Phase 4: Umsetzung (Agent 6)
  → HTML + Print-CSS fuer beide Varianten
  → Browser-PDF-Export testen
  → Bildmaterial einbinden
  → QR-Code generieren/einbinden
```
